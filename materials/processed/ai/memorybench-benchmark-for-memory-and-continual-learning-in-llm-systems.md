## Medium-Length Version

MemoryBench is a benchmark for testing whether LLM systems can improve over time by learning from user feedback. The authors argue that existing memory benchmarks mostly test static recall: give the system a long profile, conversation history, or document, then ask questions about it. That misses a more important deployment problem: can an LLM system observe how users respond to its previous answers, store useful procedural knowledge, and do better on future tasks?

The benchmark models an LLM system as having both parametric memory, such as model weights, and non-parametric memory, such as external documents, logs, profiles, or memory stores. It distinguishes declarative memory, which stores facts, from procedural memory, which stores task-improvement knowledge such as feedback, errors, preferences, and workflow lessons.

MemoryBench builds a simulation pipeline with three parts:

1. `Task Provider`: supplies user queries, task context, and evaluation metadata.
2. `User Simulator`: produces explicit and implicit user feedback on training cases.
3. `Performance Monitor`: evaluates whether the system uses context and feedback logs to improve on held-out test cases.

The benchmark covers multiple domains, languages, and task shapes. It uses 11 public datasets across open-domain, legal, and academic domains, with English and Chinese tasks and a mix of long-input/short-output, long-input/long-output, short-input/long-output, and short-input/short-output cases.

The main empirical result is cautionary. Memory support usually helps compared with a vanilla LLM, but current memory systems such as A-Mem, Mem0, and MemoryOS do not consistently beat simple RAG baselines that retrieve over task context and feedback logs. The authors argue that current systems are still weak at using procedural memory and are often inefficient as feedback accumulates.

## Full-Length Version

## Core Claim

The paper's core claim is that the field needs a benchmark for continual learning in LLM systems that treats user feedback as something to learn from, not merely as extra text to retrieve.

In MemoryBench, the unit being evaluated is not just a language model. It is a full LLM system, or `LLMsys`, with:

- a parametric memory `theta`, usually the model weights,
- a non-parametric memory `M`, such as external documents, profiles, databases, or memory entries,
- task queries `Q`,
- feedback logs `S` generated by users or the environment.

The benchmark asks whether a system can use feedback from past training interactions to reduce loss on future test queries.

## Why The Benchmark Exists

The paper starts from a scaling-law motivation. Scaling data, model parameters, and inference-time compute has been the default way to improve LLM systems, but those levers have limits. Human learners and deployed search or recommendation systems also improve by interacting with the world. That suggests a different route: systems that keep memory and learn from feedback during service time.

The authors argue that current LLM memory benchmarks are not enough because many of them are static. They test whether a system can retrieve a fact from a profile, conversation history, or long document. That is closer to reading comprehension than continual learning.

MemoryBench instead focuses on whether an LLM system can learn from feedback logs. A feedback log might say that a legal analysis used the wrong law, that a writing output was too shallow, that a user liked a response, or that a user copied the answer. These logs are not just facts about the world. They are evidence about how to do a task better next time.

## Memory Taxonomy

The paper uses a useful taxonomy.

| Type | Meaning | Example |
| --- | --- | --- |
| Declarative memory | Factual knowledge independent of task execution | A law, a user profile, a textbook fact |
| Semantic memory | User-independent declarative memory | Wikipedia-like facts, domain rules |
| Episodic memory | User-dependent declarative memory | Conversation history, user preferences |
| Procedural memory | Knowledge about how to execute tasks better | Feedback, errors, rewards, workflow corrections |

The important distinction is declarative versus procedural. Existing systems often treat all memory as declarative text. MemoryBench is designed to expose the weakness of that assumption.

## Feedback Taxonomy

The paper also divides feedback into explicit and implicit forms.

| Type | Meaning | Example |
| --- | --- | --- |
| Explicit verbose feedback | Natural-language critique | "This answer is clear but misses the penalty analysis." |
| Explicit action feedback | Direct quality action | Like, dislike |
| Implicit action feedback | Behavioral signal not explicitly framed as quality judgment | Copy, close session, refine prompt |

The benchmark simulates both verbose and action-style feedback, though the main paper results focus on off-policy experiments with explicit verbose feedback.

## Architecture Of The Benchmark

MemoryBench has three modules.

### 1. Task Provider

The task provider gathers:

- the query `q`,
- the task context `c`,
- evaluation metadata `v`.

Each task case is represented as `(q, v, c)`. Context may be optional. For example, a dialogue benchmark may need previous conversations as context, while a writing prompt may not need a long external corpus.

### 2. User Simulator

The user simulator produces feedback on training cases. For tasks with ground-truth metrics, it maps objective scores to feedback templates and action signals. For subjective or open-ended tasks, it uses an LLM-as-user setup with personas and evaluation criteria.

The simulator can produce:

- verbose natural-language critique,
- follow-up conversation turns,
- like/dislike/copy-style actions.

The authors report a human annotation study suggesting that annotators had difficulty distinguishing simulated feedback from human-written feedback in naturalness and relevance.

### 3. Performance Monitor

The performance monitor evaluates systems on held-out test data. It follows each dataset's native metrics where possible and uses an LLM-as-judge aggregation method when multiple metrics need to be merged into one score. It reports normalized aggregate performance across domains and task formats.

## Baselines

The paper compares:

- `Vanilla`: direct generation without memory,
- `BM25-S` and `BM25-M`: BM25 RAG over session-level or message-level feedback documents,
- `Embed-S` and `Embed-M`: vector RAG using Qwen3-Embedding-0.6B,
- `A-Mem`,
- `Mem0`,
- `MemoryOS`.

The main backbone is `Qwen3-8B`, with additional appendix experiments using `Qwen3-32B` and `Mistral3.2-24B`.

The main experiments are off-policy: feedback logs are generated from training cases, then systems must use those logs to improve performance on test cases. The paper also includes on-policy and action-feedback experiments in the appendix.

## Main Findings

### 1. Simulated feedback is useful

The paper finds that memory-supported systems often improve over vanilla generation, which suggests that the simulated feedback logs contain useful learning signal.

### 2. Current memory systems do not consistently beat RAG

The sharpest result is that more elaborate memory systems do not consistently outperform simple retrieval baselines. A-Mem, Mem0, and MemoryOS can help in some settings, but simple RAG over task context and feedback logs remains very competitive.

The authors interpret this as evidence that current memory systems are not general continual-learning systems. Many were built for long-context personal memory or conversational recall, not for heterogeneous procedural feedback across domains and task types.

### 3. Procedural memory is the hard part

MemoryBench gives systems both current task context and feedback logs from prior tasks. Current task context is mostly declarative. Feedback logs are procedural: they describe what went right or wrong in earlier attempts.

The paper argues that existing memory systems often treat both as generic declarative memory. That makes them weak at extracting task-improvement rules from feedback.

### 4. Efficiency is also a problem

The paper measures memory construction time and prediction time. MemoryOS often has high memory-construction cost, Mem0 can become pathologically slow in some settings, and A-Mem is more efficient but not consistently more effective than simple RAG.

This matters because continual learning systems must handle growing feedback logs. A system that improves quality but becomes too slow may not be usable in service-time learning.

## Limitations

- Feedback is simulated, not collected from real deployed LLM products.
- The main paper results emphasize off-policy explicit verbose feedback; other feedback settings are mostly in the appendix.
- LLM-as-user and LLM-as-judge components introduce model-dependent biases.
- The benchmark evaluates existing systems, but does not itself solve memory or continual learning.
- Some domains, especially legal and academic tasks, may require domain-specific interpretation of feedback.

## My Take

MemoryBench is useful because it reframes LLM memory as a service-time learning problem. The important question is not only "can the system remember this user fact?" It is "can the system notice what users corrected, turn that into procedural knowledge, and improve later behavior?"

The paper's most important empirical lesson is that current memory systems are still surprisingly shallow on this harder problem. Simple RAG remains hard to beat because it is robust, cheap, and good at retrieving relevant text. A true memory system needs to do more than store and retrieve: it needs to separate facts from feedback, infer reusable task rules, filter noise, and update itself efficiently.

## Practical Interpretation

For building agents, the benchmark suggests a few design directions:

- Store feedback separately from factual context.
- Distinguish user preferences from task-correction rules.
- Turn repeated critiques into reusable procedural memories.
- Track the provenance and reliability of feedback.
- Keep memory operations cheap enough for long-running use.
- Evaluate memory on heterogeneous tasks, not only personal QA.
